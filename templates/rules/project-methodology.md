# 项目从零到一方法论

> 适用于从想法阶段到可执行架构的全过程。按需引用。

## 总体流程

```
阶段一：需求分析（把想法变成结构化需求）
  ↓
阶段二：设计决策（逐个明确关键问题）
  ↓
阶段三：技术调研（为每个技术选型做调研对比）
  ↓
阶段四：UI 与交互设计（用户怎么用、页面怎么组织）
  ↓
阶段五：架构设计（分层、模块划分、接口边界）
  ↓
阶段六：文档体系与代码骨架（建立目录、AGENTS.md、功能域 PRD）
  ↓
阶段七：开始实现（按迭代推进，每次迭代产出可测试内容）
```

## 文档体系

### 三类文档各司其职

| 文档类型 | 位置 | 维护者 | 内容 |
|----------|------|--------|------|
| PRD | `docs/prds/` | 人 | 做什么、为什么做 |
| 技术文档 | `docs/tech/` | 人 | 怎么做、为什么这么做 |
| AGENTS.md | 跟着代码目录走 | 人 | 开发这个模块要注意什么 |
| ai2ai | `docs/ai2ai/` | AI（人审核） | 现在实际是什么样 |

### 文档职责划分

| 类别 | 位置 | 维护者 | 内容 |
|------|------|--------|------|
| me2ai | `docs/prds/`, `docs/tech/`, `AGENTS.md`, rules/ | 人 | 需求、架构、编码规范 — "应该是什么样" |
| ai2ai | `docs/ai2ai/` | AI（人审核） | 项目状态、迭代记录、经验库 — "现在实际是什么样" |

## 迭代原则

- 首次迭代必须解决：框架搭建、环境问题、从开发到测试的流程
- 每次迭代 AI 必须提供可测试的内容
- 人的角色：定规则（me2ai）+ 审任务（task）+ 跑测试（iteration 中的可测试项）
- AI 的角色：写代码 + 维护 ai2ai + 执行已审核的 task

## 阶段七：实现流程详述

### 7.1 需求澄清（每次迭代开始前，按需）

收到实现需求后不要直接写代码，先判断是否需要**需求澄清**：

**需要澄清的情况**（不跳过）：
- 需求描述模糊或开放，存在多种理解
- 涉及多种可行方案需要选择
- 需求范围可能超出一次迭代的合理规模

**可跳过的情况**：
- 用户给出了详细的实现方案和步骤
- 明确的 bug 修复，现象和预期都清晰
- task 文件已存在，只是继续执行

澄清流程：
1. **主动提问** — 对需求中模糊、矛盾、遗漏的部分提出具体问题
2. **边界探测** — 明确"做什么"和"不做什么"的边界，避免过度实现
3. **分段确认** — 如果需求复杂，将理解拆成 2-3 个段落分别确认
4. **输出设计摘要** — 澄清完成后，用一段话总结"将要做什么、怎么做、不做什么"，等用户确认后再建 task 文件

### 7.2 Git 工作流（Agent-Driven GitHub Flow）

采用 GitHub Flow + PR 门禁 + Release 分支的模式，支持多 Agent 并行开发。

**分支模型**：
- `main` — 唯一长期分支，最新开发代码
- `iter/NNN-xxx` — 迭代开发分支，从 main 创建，通过 PR 合回 main
- `release/X.Y` — 发布分支（上架时启用），从 main 切出，只接受 bugfix

**迭代分支生命周期**：
1. 从 main 创建 `iter/NNN-xxx`
2. 在分支上开发，每个子任务一次提交
3. 测试门禁通过后提 PR（`gh pr create`）
4. 人审核 PR + checklist 验收
5. 合并到 main，删除分支
6. 其他进行中的分支 `git merge main` 同步

**并行开发**：多个 AI Agent 可以同时在不同迭代分支上工作，各自提 PR，按完成顺序合并，后合并的负责解决冲突。

**为什么用 PR 而不是直接 merge**：
- PR 是每个迭代的永久记录（目标、变更、测试结果）
- GitHub diff 视图便于 Review
- 并行分支合并有序，冲突在 PR 阶段暴露
- 未来可加 CI 自动化测试门禁

**Release 工作流**（上架发布时启用）：
- 里程碑就绪时从 main 切出 `release/X.Y`
- 只做 bugfix + 版本号 + 文档，不接受新功能
- 打 tag 发布，hotfix 通过 cherry-pick 回 main

详细操作命令见 rules/ 中的 git-workflow。

### 7.3 实现顺序

对于有 UI 的项目，每个迭代内的实现顺序：
1. **逻辑层先行** — model → store → service → api，确保业务逻辑正确且有测试覆盖
2. **UI 层跟进** — 基于 UI 设计文档和已就绪的 API 实现界面
3. **端到端验证** — 打通完整链路

### 7.4 测试驱动开发（TDD）

service / store / infra 层采用**红-绿-重构**循环：

1. **红** — 先写一个描述预期行为的失败测试
2. **绿** — 写**最小**的实现代码让测试通过（不多不少）
3. **重构** — 在测试保护下优化代码结构，确保测试仍然通过
4. **提交** — 每次绿→重构完成后立即提交

核心约束：
- **YAGNI（You Aren't Gonna Need It）**— 只实现测试要求的行为，不提前预设未来需求
- 如果发现先写了实现再补测试，应停下来，删掉实现，从测试开始重来
- model 层的序列化/反序列化测试不强制 TDD，但必须有测试覆盖
- UI 层不要求 TDD，由用户验收 checklist 覆盖

### 7.5 AI 自审（任务间检查点）

每完成一个 task 后，AI 执行一次自审：

1. **规格合规** — 实现是否符合 task 文件中定义的验收标准？有无偏离？
2. **代码质量** — 是否有明显的重复代码、过度抽象、命名不当？
3. **测试覆盖** — 新增的公开 API 是否都有测试？测试是否覆盖了异常路径？
4. **回归影响** — 改动是否可能影响已有功能？如果是，对应的回归测试是否存在？

自审结果不需要单独输出文档，但如果发现问题应立即修复后再标记 task 为完成。

### 7.6 子代理驱动开发（可选）

当迭代包含多个**互相独立**的任务时，可以使用子代理并行执行：

- 每个子代理处理一个独立任务，自带完整的上下文（task 描述、相关文件、验证标准）
- 主代理负责分发任务、收集结果、执行 review
- 子代理完成后，主代理检查：(1) 是否符合 task 规格 (2) 代码质量是否达标
- 不合格的任务退回重做，不是补丁修修

适用场景：3 个以上独立任务、任务间无代码依赖
不适用场景：任务有顺序依赖、涉及同一模块的修改
