# 迭代工作流

ai2ai 迭代的完整操作手册。根据当前所处阶段执行对应步骤。

---

## 0. 需求澄清（按需）

收到需求后先判断是否需要澄清：

**需要澄清**：需求模糊、存在多种理解、涉及方案选择、范围可能超出一次迭代。

**可跳过**：用户给出了详细步骤、明确的 bug 修复、task 文件已存在只是继续执行。

澄清流程：
1. 提出具体问题（模糊、矛盾、遗漏的部分）
2. 明确"做什么"和"不做什么"（YAGNI）
3. 输出设计摘要，等用户确认后再进入步骤 1

---

## 1. 迭代开始：创建分支 + task 文件

```bash
git checkout main
git checkout -b iter/NNN-xxx
```

创建 `docs/ai2ai/tasks/NNN-xxx.md`：

```markdown
# 任务 NNN: [标题]

> 状态：🔨 进行中
> 分支：iter/NNN-xxx
> 配对迭代：[iterations/NNN-xxx.md](../iterations/NNN-xxx.md)

## 迭代目标
[一句话描述]

## 验收标准（人审核/补充）
- 能做到 XX
- 能看到 YY
- ZZ 功能不受影响

## 任务分解
- [ ] T1: [描述]
  - 验证：[具体验证标准]
- [ ] T2: ...
```

任务分解原则：原子单元、有明确验证标准、按依赖顺序、3-8 个任务、YAGNI。

---

## 2. 迭代进行中：TDD + 自审

### TDD 循环（service / store / infra 层）

1. **红** — 先写描述预期行为的失败测试
2. **绿** — 写最小实现让测试通过
3. **重构** — 在测试保护下优化结构
4. **提交** — 每次绿→重构完成后立即提交

约束：只实现测试要求的行为；发现先写实现再补测试时，删掉实现从测试重来。

### 每个 task 完成后自审

标记 `[x]` 前检查：
1. **规格合规** — 是否符合 task 验收标准？
2. **代码质量** — 有无重复代码、过度抽象、命名不当？
3. **测试覆盖** — 新增公开 API 是否都有测试？是否覆盖异常路径？
4. **回归影响** — 改动是否可能影响已有功能？

发现问题立即修复，修复后再标记完成。

### 子代理并行（可选）

3 个以上互相独立的任务时可并行：每个子代理处理一个任务，主代理分发+收集+两轮 review（规格合规、代码质量），不合格退回重做。

---

## 3. 迭代结束：归档

测试门禁：全量测试通过后才进入归档。

### Commit 时机规则

- **实现代码**：每个 task 完成（绿→重构）后立即 commit
- **归档文档**：等用户验收通过后统一一次性 commit

> 原因：验收失败时需要修改归档文档，提前 commit 会产生多余的 revert，污染 git 历史。

### 归档步骤

1. 运行 `/checklist` 生成验收清单
2. **task 文件**：状态改为 `✅ 已关闭`
3. **新建 `docs/ai2ai/iterations/NNN-xxx.md`**：
   - 记录：目标、完成内容、未完成/遗留、回归影响
   - "测试结果"从 checklist.md 勾选结果归档
4. **更新 `docs/ai2ai/status.md`**：最后更新日期、当前迭代指向、模块就绪度、端到端联通状态、已知问题
5. **按需更新 `docs/ai2ai/decisions.md`**
6. **等用户验收通过后**：`git add docs/ai2ai/ && git commit -m "docs: NNN 验收通过，更新归档和 checklist"`
7. **提交 PR**，合并到 main

---

## Checklist 维护规则

文件：`docs/ai2ai/checklist.md`（唯一活的用户验收清单）

### 格式

```markdown
# 测试 Checklist

> 当前迭代：NNN-xxx

## 本迭代新增
- [ ] [测试项]：[操作步骤] → [预期结果]

## 回归测试
- [ ] [测试项]：[操作步骤] → [预期结果]
```

### 生成规则

- **本迭代新增**：从 task 验收标准展开为具体操作步骤 + 预期结果
- **回归测试**：继承上一迭代通过的测试项 + `test-suite.md` 中的全局用例（去重）
- 上一迭代失败项：本迭代修复了 → 移到"本迭代新增"；未修复 → 保留并标注 `⚠️ 上次失败`

### 失败项处理

```markdown
- ❌ [测试项描述]
  > 现象：...
  > 备注：...
```

| 类型 | 判断 | 处理 |
|------|------|------|
| 当轮修复 | 属于本迭代验收标准范围 | AI 修复 → 改回 `[ ]` → 人重新测试 |
| 下轮修复 | 超出本迭代范围 | 标注 `→ 转入下一迭代`，下一迭代自动生成修复任务 |

安全护栏：同一项连续修复 3 次仍未通过 → 转为下轮修复并报告。

### 归档规则

新迭代开始时：将当前 checklist.md 完整内容（含勾选状态）复制到 `iterations/NNN-xxx.md` 的"测试结果"章节，然后重置为新迭代内容。

---

## 全局用例库

`docs/ai2ai/test-suite.md`：经验证的长期回归用例，只增不减。

归档时沉淀条件：
- 覆盖核心链路的端到端测试
- 连续 3 次以上迭代作为回归项通过
- 曾经失败后修复的测试

不沉淀：一次性功能验证、纯 UI 样式测试。

---

## 模块状态标记

| 标记 | 含义 |
|------|------|
| ✅ 可用 | 功能已实现且可正常使用 |
| 🔨 部分 | 部分功能已实现 |
| 🔨 骨架 | 仅有接口/结构定义，无实际逻辑 |
| ❌ 空壳 | 文件存在但无实现 |
| ❌ 未开始 | 尚未创建 |

## decisions.md 记录类型

| 前缀 | 类型 |
|------|------|
| D | 技术微决策（设计文档未覆盖的实现选型） |
| L | 踩坑记录（技术限制、意外行为、绕过方案） |
| B | Boundary 变更（AI 决策边界规则的新增/调整） |

---

## 安全护栏

- **错误循环**：连续 3 次相同修复未成功 → 停止，报告现象 + 已尝试方案 + 可能原因，等待人工介入
- **变更范围**：单次迭代修改文件超过 10 个 → 暂停确认是否需要拆分迭代
