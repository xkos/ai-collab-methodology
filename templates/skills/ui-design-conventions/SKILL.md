# UI 设计规范（通用模板）

> 框架无关的 UI 设计规范骨架。请根据你的技术栈（Flutter/SwiftUI/Electron/React 等）填充具体内容。
> 与 `ui-prototype` 规则配合使用：原型阶段用 `ui-prototype` 验证布局，实现阶段用本 skill 指导代码质量。

生成有辨识度、生产级的界面，避免千篇一律的"AI 味"设计。

## Design Thinking

写代码前，先确定美学方向：

- **Purpose** — 这个界面解决什么问题？用户是谁？使用频率如何？
- **Tone** — 选择一个明确的风格立场（极简克制、信息密集、温暖有机、冷峻工业、杂志编辑风等）。
- **Constraints** — 平台约束、性能要求、无障碍需求。
- **Differentiation** — 什么让这个界面令人印象深刻？一个记忆点就够。

核心原则：大胆极繁和精致极简都行，关键是 intentionality — 每个设计决策都有理由。

## 主题系统（唯一真相源）

<!-- 根据你的框架填充。核心思想：样式定义集中在主题层，组件层只消费、不定义。 -->

### 核心规则

- 色彩通过主题系统的语义化 token 引用，不在组件中硬编码色值
- 文字样式通过主题的排版层级引用，不在组件中手写字号/字重
- 组件默认样式在主题层统一配置，组件处零配置直接使用
- 组件层只做局部覆盖 — 仅当个别组件确实需要偏离全局默认时
- 项目特有的语义化 token 通过主题扩展机制定义，不用全局常量类

### 禁止

- 色值散落在组件代码中
- 在组件中手写完整的文字样式而不引用主题排版
- 每个组件单独设置样式，重复定义相同配置
- 用全局静态常量类替代主题系统

## 美学规范

### 色彩

- 主色调搭配锐利强调色，优于均匀分布的多色方案
- 善用透明度和表面色调创造层次感
- 暗色/亮色主题都要精心设计，不是简单反转

### 字体与排版

- 选择有性格的字体对：标题字体 + 正文字体
- 建立清晰的字体层级（display → title → body → label）
- 中文场景考虑有辨识度的字体选择（思源黑体/宋体、霞鹜文楷等）
- 西文场景避免系统默认字体，选择有特色的字体
- 行高、字间距、段落间距需要精心调教

### 空间与布局

- 统一间距体系（如 4/8/12/16/24/32/48）
- 不对称布局、错落排列比死板网格更有生气
- 负空间（留白）是设计元素，不是浪费
- 根据目标平台选择合适的布局模式（桌面多栏、移动单栏等）

### 动效与交互

- 用隐式/声明式动画处理简单过渡
- 页面切换有连贯的过渡效果
- 列表项出现用错位延迟制造节奏感
- 交互反馈要细腻：微妙的阴影变化、背景色过渡、缩放

### 视觉细节与质感

- 背景避免纯色：考虑微妙渐变、噪点纹理、模糊效果
- 选择有性格的图标集，不局限于框架默认图标
- 圆角要统一且有意图：全圆角(柔和) vs 小圆角(锐利) vs 混合(层次)
- 分割线、边框用于结构化，但不要过度使用

## 平台特有考量

### 桌面端

- 充分利用宽屏空间：多栏布局、侧边栏+内容区、Master-Detail
- 更高的信息密度，不需要像移动端那样大留白大按钮
- 键鼠优先：hover/focus 状态、右键菜单、键盘快捷键、拖拽
- 考虑平台原生融合（标题栏、毛玻璃效果等）
- 支持窗口缩放时的响应式布局断点

### 移动端

- 触摸优先：足够大的点击区域（最小 44pt）、手势操作
- 单手可达性：关键操作放在屏幕下半部分
- 适配不同屏幕尺寸和安全区域
- 考虑系统级交互模式（下拉刷新、滑动返回等）

## 实现要求

- 所有代码必须可运行、无编译错误
- 组件拆分合理，单个构建方法不过长
- 颜色、间距、字体通过主题系统管理，不硬编码
- 动画时长控制在合理范围（150ms-400ms）
- 每次生成的设计应该有所不同 — 轮换明暗主题、字体、配色方案

## 反模式清单

生成 UI 时自检，以下任何一项出现即需修正：

- [ ] 使用了框架默认主题未做定制
- [ ] 全部使用系统默认字体
- [ ] 可点击/可交互元素无反馈（hover/press）
- [ ] padding/margin 数值随意不统一
- [ ] 颜色硬编码散落在各组件中
- [ ] 背景纯色无质感处理
- [ ] 动画生硬或缺失
- [ ] 桌面端使用手机式单栏布局 / 移动端使用桌面式密集布局
- [ ] 关键状态缺失（空状态、加载态、错误态）
