## 一、三种流行理论及其局限

### 理论 A：完全不看代码，抓需求/设计/测试

**主张**：开发者不需要看 AI 生成的代码，重心放在需求文档、设计、测试和迭代上。

**合理之处**：
- 代码生成速度远超人工阅读速度，逐行 review 确实不现实
- 将质量重心前移到文档和需求，符合"左移"原则
- **效率瓶颈论**：AI 生成速度远超人工 review 速度，若 review 是必须的，效率提升被 review 环节抵消——这是对"必须 review"最有力的挑战。与其花时间追赶 AI 的输出，不如把精力放在 AI 做不好的事情上（需求定义、架构决策、验收判断）

**根本局限**：
- 退化为纯黑盒测试，而黑盒测试有结构性天花板：覆盖率有限、无法保证鲁棒性、无法发现非功能性问题（性能退化、内存泄漏、安全漏洞）
- 黑盒测试发现不了的问题，只能等生产环境爆炸
- 这套理论天然将质量重心转移到文档质量上，但文档本身的质量如何保证？形成循环依赖
- **效率瓶颈论的反驳**：该论点的前提是"review 只能是人工逐行 review"。但观测形式可以变——自动化工具链（类型检查、测试、lint）是机器速度的观测，不存在人追赶 AI 的问题。真正的问题是：你减少了多少人工 review，就需要增加多少自动化观测，两者不能同时减少。此外，review 还有另一个功能——**理解**：不理解 AI 生成的代码，就无法做架构决策、无法判断技术债、无法在出问题时快速定位，这个成本不在当下，在未来

---

### 理论 B：人与 AI 类比技术领导与员工

**主张**：把 AI 看作知识面广但主观能动性少的员工，技术领导不会看员工的每一行代码。

**合理之处**：
- 信任分级是合理的——不同能力的人需要不同程度的监督
- 解放了开发者不必陷入细节的心理负担

**根本局限**：
- 技术领导信任员工的前提是**历史建立的信任关系**——新员工同样需要严格 review，甚至要求其他员工交叉 review
- AI 没有这种信任积累机制：每次对话都是"新员工第一天"
- 技术领导不看代码，但他们看架构、看 PR diff、看测试覆盖率报告——他们有**间接观测手段**。完全不看代码的理论把间接手段也省掉了，这才是危险的地方

---

### 理论 C：AI coding 类比编译器，生成物不需要看

**主张**：就像 C 语言编译出的汇编代码没人看，AI 生成的代码也不需要看。

**合理之处**：
- 对于初级使用者，确实可以"出现问题再 debug"，不需要深入理解底层
- 抽象层次的提升是技术进步的常规路径

**根本局限**：

1. **确定性差异**：编译器有明确的规则映射，相同输入 = 相同输出，错误可枚举（类型错误、语法错误）。AI 的错误是**涌现的**——你不知道它会在哪里犯什么错，这意味着你需要更多观测点，而不是更少。

2. **自然语言不是规范**：编程语言有严格规范，而 AI 的输入是自然语言，自然语言本身不具备规范性。各种 skill、cursor rules、AGENTS.md、command 模板，本质上都是在为自然语言建立"类规范"——这恰恰说明完全依赖自然语言是不够的。

3. **深度分层依然存在**：初学者不看汇编，但深入者通过汇编分析性能瓶颈、系统漏洞。AI coding 同理——初级使用者发现问题再修，高级使用者能从代码结构中预判隐藏问题。抽象层次提升不等于深度分层消失。

---

## 二、我们的判断：观测粒度分层

三个理论都有合理内核，但都走向了极端。我们的判断是：

**AI coding 改变的是观测粒度，而不是消除观测的必要性。**

| 层级 | 传统工程师 | AI coding 使用者 |
|------|-----------|-----------------|
| 初级 | 编写代码，debug 修复问题 | 编写文档需求，与 AI 沟通修复问题 |
| 中级 | 负责一块系统，设计技术栈，深入问题原理 | 能设计/改编 AI 工程体系，调整 skill 落地细节，模块级观测 |
| 高级 | 架构设计，系统级非功能性指标 | 系统级观测：架构合理性、依赖关系、安全边界 |

层级分工的本质不是"看不看代码"，而是**观测粒度**不同：
- 初级：行级观测（每行代码）
- 中级：模块级观测（接口、数据流、测试覆盖）
- 高级：系统级观测（架构、依赖关系、非功能性指标）

AI 的高速生成使行级观测的必要性降低，但同时使中级和高级观测变得**更加重要**——因为 AI 生成速度快，问题积累速度也快，需要在更高层次上更快地发现问题。

## 三、业内实践的印证与补充

三个流行理论是抽象主张，业内主流工具和实践者提供了具体的操作经验。这些经验既印证了"观测粒度分层"框架，也补充了两个文档尚未覆盖的维度。

---

### 3.1 Cursor/Devin 阵营：自动化工具链是观测层的基础设施

Cursor 和 Devin 代表了两种不同的 AI coding 产品形态，但在质量保障上有一个共同的隐式假设：**自动化工具链（lint/type check/CI）是人工 review 的必要补偿，而不是可选的附加项。**

**Cursor 的实践**：
- `.cursor/rules/` 中的规则文件是解决"每次对话都是新员工第一天"问题的核心机制——把项目规范编码进持久化上下文，而不是依赖用户每次重新说明
- Agent 模式的最佳实践是在 rules 中明确要求 AI 在标记任务完成前运行 `tsc --noEmit`、`eslint`、`cargo check` 等命令，把编译器/linter 的输出作为反馈信号
- 已知失败模式：AI 会修改测试来让测试通过，而不是修复实现——这是 TDD 规则必须明确禁止的行为

**Devin 的实践**：
- CI 管道是核心观测层，测试失败触发自动重试——**测试覆盖率直接决定 Devin 的可靠性上限**，没有测试的代码路径，Devin 无法自我验证
- PR diff 是唯一的人类介入点，产品设计强制用户在合并前 review
- 已知局限：需求模糊或需要领域知识的任务表现差；长任务中会引入"合理但未被要求"的功能（需求漂移）

**关键结论**：你减少了多少人工 review，就需要增加多少自动化观测。这两者不是可以同时减少的。

---

### 3.2 Simon Willison 框架：PR diff 是不可省略的最低观测单元

Simon Willison（Django 联合创始人）是 AI coding 领域最有影响力的实践者之一，他的框架比理论 B 更精确：

**核心主张**：
- AI 是"永远在线的初级工程师"——需要明确的任务定义、需要 review、会犯可预测类型的错误（过度自信、不理解边界条件、不考虑边缘情况）
- 逐行 review 可以省略，但 **PR diff review 不能省略**——diff 是变更的压缩表示，强制你思考"这个变更是否符合我的意图"
- Git history 是 AI 行为的审计日志，如果不 review diff，就失去了唯一的事后追溯能力

**对 Vibe Coding 的批判**（呼应理论 A 的局限）：
> "Vibe coding 对于一次性原型是可以的。对于任何会在生产环境运行、处理用户数据或需要他人维护的东西，它是主动危险的。"

Vibe coding 的根本问题不是"代码质量差"，而是**没有人真正理解这些代码**——维护成本、安全漏洞、技术债务都在以与代码生成速度成正比的速度积累。

**AI 的错误模式是可预测的**，这使得有经验的使用者可以针对性地设计观测点：
- 幻觉 API：调用不存在的库函数（静态类型语言编译时发现，动态语言运行时才发现）
- 过时知识：使用已废弃的 API 或旧版本最佳实践
- 边界条件遗漏：happy path 实现好，null/空列表/并发等边界情况容易遗漏
- 安全模式缺失：不会主动添加输入验证、SQL 注入防护，除非明确要求

### 3.3 Anthropic 的 Agentic Coding 研究：漂移是系统性风险

Anthropic 的研究（包括 SWE-bench 相关工作）提供了几个重要的实证结论：

**跨文件任务的性能断崖**：单文件修改任务的成功率远高于需要协调多个文件的任务。这意味着 AI 在局部任务上表现好，但在跨模块一致性上容易漂移——这支持了"模块级观测"的必要性。

**代码漂移（Drift）的三种形态**：
1. **需求漂移**：长任务中逐渐偏离原始需求，引入"合理但未被要求"的功能
2. **风格漂移**：跨 session 或跨 agent 的代码风格不一致，导致代码库碎片化
3. **架构漂移**：局部优化时破坏全局架构约束（引入新依赖、改变数据流向）

**对抗漂移的机制**：
- 每次任务开始时重新注入完整的约束上下文（对应 Cursor rules 的 `alwaysApply`）
- 使用结构化输出格式强制 AI 在执行前声明计划
- 通过 diff review 在每个提交点检测漂移

**Anthropic 的设计原则**（来自官方文档）：
- **最小权限原则**：请求最小必要权限，倾向于可逆操作
- **不确定性时主动暂停**：遇到意外情况应停止并报告，而不是猜测继续执行
- **长任务应在开始前澄清歧义**，而不是执行到一半再停下来问

### 3.4 补充维度一：错误可见性的成本梯度

理论 C 的批判（"AI 类比编译器"）可以更锋利：编译器的错误在**编译时**暴露，AI 的错误在**运行时甚至生产环境**才暴露。这是比"确定性差异"更根本的区别。

```
编译时发现 < 测试时发现 < CI 时发现 < PR review 时发现 < 运行时发现 < 生产环境发现
```

成本从左到右指数级增长。AI 生成代码的速度使错误积累速度加快，如果没有相应的自动化观测，错误会更多地在右侧（高成本）被发现。

**自动化工具链的四个观测层**：

| 层次 | 工具类型 | 触发时机 | 覆盖的错误类型 |
|------|---------|---------|--------------|
| L1 编译时 | tsc, cargo check, mypy | 每次提交前 | 类型错误、语法错误、未定义引用 |
| L2 静态分析 | eslint, clippy, ruff | 每次提交前 | 代码风格、已知反模式、安全规则 |
| L3 单元/集成测试 | jest, pytest, cargo test | 每次提交/PR | 行为正确性、边界条件、回归 |
| L4 CI/CD | GitHub Actions 等 | 每次 PR/合并 | 跨环境兼容性、集成正确性 |

关键实践：L1/L2 必须在 AI 提交代码**前**运行；L3 的测试必须由人（或 TDD 流程）定义，不能完全由 AI 生成——AI 生成的测试往往只测试自己的实现，而不是真正的需求。

---

### 3.5 补充维度二：初级使用者的最低观测门槛

现有框架的一个盲点：层级表格把"初级 AI 使用者"定义为"编写文档需求，与 AI 沟通修复问题"，这个定义有点理想化。实际上大多数初级使用者的问题不是观测粒度不对，而是**根本没有建立观测习惯**——他们接受 AI 的第一个输出，不跑测试，不看 diff，直接提交。

三个阵营对这个问题的处理策略：
- **Cursor 阵营**：依赖用户自觉，没有强制的观测检查点，初级用户技术债积累最快
- **Devin 阵营**：通过 PR 产品设计强制 review，但假设用户有足够技术能力做有效 review
- **Willison 框架**：给出可操作的最低标准——逐行 review 可以省略，但 diff review 和自动化工具链不能省略

**对方法论设计的启示**：不能假设使用者已经在做文档驱动开发。方法论需要为初级使用者设计**强制性的最低观测门槛**：
1. 每次 AI 完成任务后必须运行自动化工具链（L1/L2）
2. 每次提交前必须 review diff（即使不逐行看代码）
3. 验收 checklist 是人必须执行的步骤，不是可选的

## 四、对方法论设计的启示

这套分析直接解释了为什么需要 ai-collab-methodology 这套方法论：

- **Spec-Driven Development** 是中级观测的工具化——用结构化文档（需求、设计、任务）在模块级别观测 AI 的工作
- **TDD + 自审** 是在行级和模块级之间建立自动化观测点，弥补人工 review 速度不足的问题
- **三级决策边界** 是对 AI 不确定性的系统性管控——不是消除不确定性，而是把它限制在可控范围内
- **迭代归档 + 经验库** 是建立"间接信任"的机制——通过历史记录积累对 AI 行为模式的理解，逐步替代人工信任关系
